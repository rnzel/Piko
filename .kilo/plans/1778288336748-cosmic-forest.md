# Groups Feature Analysis: Debugging Persisting/Saving Issues

## Executive Summary

This analysis traces the complete Group creation flow from UI to Firestore and identifies the root causes of two primary issues:

1. **Groups not persisting to Firebase/Firestore** - Multiple failure points identified
2. **User search not working during member invitation** - Intentionally mocked, not connected to Firebase

---

## Part 1: Why Groups Are Not Saving to Firebase

### Flow Analysis

```
GroupsScreen → groupService.createGroup() → syncOrchestrator.addGroup() → Firestore
```

### Root Cause #1: AuthContext State Check Issue

**File:** `contexts/AuthContext.tsx` (lines 76-81)

```javascript
useEffect(() => {
  const unsubscribe = syncOrchestrator.subscribe((state) => {
    const syncState = syncOrchestrator.syncState;
    setSyncState(syncState);
  });
  return () => { unsubscribe(); };
}, []);
```

**Problem:** The `AuthContext` subscribes to orchestrator state changes, but the subscription callback doesn't use `setState` directly. The pattern:
```javascript
setSyncState(syncOrchestrator.syncState)  // Reading from orchestrator, not using the emitted state
```

This means when `SyncOrchestrator` emits state changes, the `AuthContext`'s `setSyncState` is called, which triggers re-renders. However, the critical issue is that the `_uid` in `SyncOrchestrator` might not be set correctly during group creation.

### Root Cause #2: `_canWriteThrough()` Returns False

**File:** `services/SyncOrchestrator.ts` (lines 710-716)

```javascript
private _canWriteThrough(): boolean {
  return (
    (this._state === SyncLifecycleState.READY ||
      this._state === SyncLifecycleState.REALTIME_READY) &&
    this._uid !== null
  );
}
```

**Problem:** The `addGroup()` method (line 829-869) calls `_canWriteThrough()` before writing to Firestore. If the sync state is not `READY` or `REALTIME_READY`, the Firestore write is SKIPPED entirely with only a console error.

**Critical Path:** When a user creates a group immediately after signing in, the `SyncOrchestrator` may still be in one of these states:
- `UPLOADING_LOCAL` (line 223)
- `HYDRATING` (line 251)
- Not yet initialized at all

### Root Cause #3: No Firestore Error Propagation

**File:** `services/SyncOrchestrator.ts` (lines 863-868)

```javascript
} catch (e) {
  console.error(`[SyncOrchestrator] addGroup firestore error:`, e);
}
```

**Problem:** Firestore write errors are caught but only logged to console. The `groupService.createGroup()` method doesn't await or check for Firestore success. The group appears "created" locally because `groupStorage.addGroup(group)` succeeds, but the remote write may have failed silently.

### Root Cause #4: Group Member Creation Not Tied to Creator UID

**File:** `services/groupService.ts` (lines 105-112)

```javascript
const creatorMember: GroupMember = {
  uid: createdBy,
  email: "",      // Empty - should come from userProfile
  displayName: "", // Empty - should come from userProfile
  role: "owner",
  joinedAt: Date.now(),
  notificationsEnabled: true,
};
```

**Problem:** The creator's member record is created with empty `email` and `displayName` fields. While not causing persistence issues directly, this creates incomplete data in Firestore.

## Part 2: User Search During Member Invitation

### The Mock Implementation Problem

**File:** `components/groups/MemberSearch.tsx` (lines 26-44)

```javascript
const mockSearchUsers = async (query: string): Promise<UserResult[]> => {
  await new Promise((resolve) => setTimeout(resolve, 500));
  if (!query.trim()) return [];

  const mockUsers: UserResult[] = [
    { id: "1", email: "alex@example.com", displayName: "Alex Rivera" },
    // ... hardcoded users
  ];
  // ... filter logic
};
```

**Critical Finding:** The member search uses a **hardcoded mock function** that returns static test users. There is NO Firebase integration for searching users by email or display name.

### Missing Backend Functionality

The following components are MISSING for real user search:

1. **No users collection query** - There's no Firestore query to search `users` collection by email/displayName
2. **No user search service** - No backend endpoint/function to query users
3. **No security rules for user discovery** - Current rules only allow users to read their own document:
   ```javascript
   // firebase.rules line 8: Only the user can read/write their own document
   allow read, write: if request.auth != null && request.auth.uid == userId;
   ```

## Firestore Rules Analysis

**File:** `firebase.rules`

### Issue #1: Group Creation Rules (lines 56-58)

```javascript
allow create: if request.auth != null
              && request.resource.data.createdBy == request.auth.uid
              && request.resource.data.memberCount == 1;
```

**Problem:** Rules require `memberCount == 1` on create, but `addGroup()` in `SyncOrchestrator` sets `memberCount: 1` in the sanitized data. This should work, but if `memberCount` is missing or different due to normalization, the write fails.

### Issue #2: No User Discovery Capability

The security rules do not provide any mechanism for:
- Searching users by email
- Searching users by displayName
- Listing users for invitation purposes

## Data Flow Issues

### Storage Layer Problem

**File:** `services/storageService.ts`

The storage service uses TWO different storage locations:

1. **Global storage** (lines 93-260): `STORAGE_KEYS.GROUPS` - `@piko_groups`
2. **User-scoped storage** (lines 347-395): `userGroupsKey(uid)` - `@piko_user_groups_${uid}`

**Problem:** `groupStorage.getGroups()` reads from the global key (`@piko_groups`), but after migration, groups may be stored in the user-scoped key. This disconnect causes groups to "disappear" when switching between states.

## Recommended Fixes

### Fix 1: Ensure SyncOrchestrator is READY before group creation

In `app/(tabs)/groups.tsx`, add a state check before allowing group creation:

```javascript
if (syncState !== SyncState.READY) {
  Alert.alert("Not ready", "Please wait for sync to complete before creating groups.");
  return;
}
```

### Fix 2: Implement real user search in MemberSearch.tsx

Replace the mock function with a real Firestore query:

```javascript
const searchUsers = async (query: string): Promise<UserResult[]> => {
  if (!query.trim()) return [];
  
  // Query users collection where email or displayName matches
  const usersRef = collection(db, "users");
  const q = query(
    usersRef,
    or(
      where("email", ">=", query.toLowerCase()),
      where("displayName", ">=", query)
    ),
    limit(20)
  );
  // ... return results
};
```

### Fix 3: Update Firestore security rules for user search

```javascript
// Add a users search rule
match /users/{document=**} {
  allow list: if request.auth != null; // For searching users by email/displayName
}
```

### Fix 4: Add proper error feedback in groupService.createGroup()

```javascript
const result = await syncOrchestrator.addGroup(newGroup, creatorMember);
if (!result.success) {
  throw new Error(result.error || "Failed to create group");
}
```
